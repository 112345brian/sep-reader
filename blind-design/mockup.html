Content Delivery

Updated 2 h ago · 4 min
3 notes

The content delivery layer handles all asset serving for the application. It sits in front of the Origin Servers and provides edge caching at PoPs across 12 regions. See Cache Invalidation Strategy for the rules governing TTLs.

Check whether the 12-region claim is still current — last I heard we were expanding to APAC.

Architecture overview

Requests arrive at the Global Load Balancer, which routes to the nearest healthy PoP. If the requested asset is cached (hit ratio ~94%), it is served directly. On a miss, the PoP fetches from origin and caches the response per the Caching Policy.

The Auth Service issues short-lived signed URLs for private assets. These tokens embed the user ID and expiry; the edge validates them without an origin roundtrip.

Configuration

The main knobs in cdn.config.ts:

  • TTL — default 3600 s; overridable per content type
  • Vary headers — always vary on Accept-Encoding
  • Stale-while-revalidate — 60 s for HTML, 0 for JSON
The CDN is the first line of defense for traffic spikes. During the March incident it absorbed 40x normal load with no latency increase.

Known gaps

Streaming responses currently bypass the CDN — the Streaming Cache RFC is tracking the design. Image optimization (Image Pipeline) is also not yet hooked in.
